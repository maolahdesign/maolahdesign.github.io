這份講義怎麼用？

有沒有組過那種附說明書的模型？零件一包一包編好號，說明書一頁一頁照著做，做完就是盒子上那台。這份講義就是照這個邏輯寫的：每一節先跟你聊這一節要解決什麼問題，接著介紹會用到的工具（規格、價錢、地雷都寫了），然後給你一步一步照著做的操作步驟與完整範例；卡關的時候先翻該節的「常見錯誤與排除」，你會發現你卡的地方，大家都卡過；做完再用「自我檢核」確認自己不是「做完了」，而是「學會了」。

優勢劣勢看自己，機會威脅看外面；自己的能改，外面的只能應對。

SWOT 最常見的死法是寫得太空泛：「劣勢：資源不足」——誰的資源充足過？「優勢：新鮮食材」——誰家用不新鮮的？每一格都要寫得出具體證據，指得出「跟誰比、差在哪」，這張表才有用；寫不出證據的那一格，你就該回去做功課。四個格子填完，策略其實已經自己浮出來了：用優勢去接機會，補劣勢去擋威脅。是不是很簡單？簡單到常常被亂用（哈哈）。動手填一次就知道了，卡住歡迎來討論，在線等。

1-2｜使用者研究自動化與數據合成

訪談做完了，然後呢？很多人的做法是把逐字稿直接丟給 AI：「幫我生成人物誌」，三十秒後拿到一份看起來超專業的文件，開心拿去報告，被利害關係人問一句「這個痛點是哪位受訪者說的？」，當場安靜。這一節就是要避免那個安靜的瞬間：你會搞懂 Granola、Dovetail、Claude／Gemini 三個工具為什麼要分工，實際走完「錄音 → 逐字稿 → 標籤化 → 生成人物誌」的完整流程，並學會檢查 AI 生成的每一項特徵，是不是都指得回受訪者的原話。

核心內容：為什麼需要三種工具，而不是直接讓 Claude 做完全部

Granola 解決的是「擷取」問題：不需要邀請 bot 加入 Zoom/Teams 通話，而是直接錄系統音訊、本機即時轉錄，跨平台通用（實體訪談、任何會議軟體都能用），並有專門的 User Interview 範本，錄完直接產出結構化訪談筆記。少了這一層，你得自己想辦法把訪談錄音轉成乾淨逐字稿，才能餵給 Claude 或 Gemini。

Dovetail 解決的是「治理與存證」問題：它是專門的研究資料庫，每一個洞察、每一個人物誌特徵都能回溯連結到原始逐字稿的確切時間點與引言。這在跟利害關係人報告時很關鍵——你能證明「這個洞察來自哪三位受訪者的哪句話」，而不是一個看起來合理、但其實是模型自己整併出來的說法。Claude/Gemini 是無狀態的生成引擎，可以叫它讀一堆逐字稿去生成人物誌，但它不會自動幫你維護一個「所有洞察都可追溯回原始引言」的長期結構化資料庫。

Claude 或 Gemini 則接手最後一步：讀取 Dovetail 裡已經整理好、有證據支撐的主題，生成動態人物誌、旅程地圖與洞察報告。整條邏輯是：Granola 負責把訪談變乾淨、結構化的逐字稿；Dovetail 負責把多場訪談的引言標籤化、聚類、留下可追溯的證據鏈；Claude 或 Gemini 負責生成最後的敘事產出。少了前兩層，生成出來的東西會失去「這是真的使用者說的話」這個可信度基礎——這正是第一份教案中「使用者信任」概念在研究階段的具體體現：一份無法回溯來源的 AI 生成人物誌，利害關係人有充分理由不信任它。

第二週：提示詞工程與快速原型

2-1｜從文字到線框圖

開始大量生成之前，先搞懂兩個概念：「逐字預測」跟「上下文視窗」。聽起來很理論對吧？但你之後每天都會撞到它們——為什麼同一句 prompt 每次結果都不一樣？為什麼換個工具它就忘了你剛剛說過的話？答案都在這兩個概念裡。搞懂之後，你會用 Google Stitch 把研究洞察變成多個線框圖方向，再讓 Claude 把選定的方向變成可運作的雛形。

核心內容：生成式 AI 原理速成

玩過文字接龍嗎？大型語言模型做的事就是接龍：持續預測「接下來最可能出現的字（token）」。它不是先想好完整答案再打出來，是邊生成邊決定下一步——所以 AI 有時會講出看似合理、其實錯誤的內容（幻覺），因為它給你的是「統計上說得通的接續」，不是查證過的事實。這也解釋了為什麼同一句 prompt 在 Stitch 或 Claude 上、甚至同一個工具重跑一次，結果都可能不同：生成本質上是機率性的，跟 Figma 靜態稿「同樣操作、永遠同樣結果」是兩個世界。

再來是「上下文視窗」。把每個 AI 工具想成一個只看得到眼前白板的同事：白板上寫了什麼它就知道什麼；白板放不下的、寫在別間會議室的，它一概不知道。所以當你把 Stitch 生成的方向丟給 Claude 接手，Claude 的白板是空白的——它不知道你在 Stitch 設定過什麼限制，你得把「目標使用者」「品牌調性」「這是延續哪一版方向」重新寫上去；不然產出偏題別怪它，它是真的沒看到。

另外一個實務上重要的區分：目前落地應用大致分三種型態——純生成（直接依模型內建知識生成，需標示「這是草稿」）、檢索增強生成 RAG（先查外部資料再生成，需附來源）、代理式 AI（自主規劃多步驟並呼叫工具，需要人工核可節點）。這一階段做的 Text-to-Wireframe 屬於「純生成」，產出是草稿而非定論，仍需人工篩選與修正。

核心內容：使用者信任與可解釋性

使用者對 AI 系統的信任不是單一開關，而是隨互動經驗動態變化的過程，其中「AI 犯錯之後」的那一次互動，往往是信任是否能修復的關鍵節點。設計 Agent 時特別容易遇到兩種相反卻同樣常見的失衡：

演算法反感：即使證據顯示 AI 表現優於人類，使用者仍傾向拒絕採用、轉而依賴人類判斷。成因包括對錯誤的容忍度不對等（看到 AI 犯錯後信任下降幅度更大）、缺乏掌控感、輸出結果難以理解。研究發現一個緩解方向：當使用者被允許對 AI 的建議做出微幅調整，演算法反感的程度會明顯降低——這代表「保留使用者的最終控制權」是有效的設計槓桿。

過度依賴悖論：使用者即使面對錯誤的 AI 建議，仍然選擇採信。直覺上，若 AI 提供解釋，使用者應該更能辨別建議是否可信；但研究發現單純附加解釋並不必然降低過度依賴，尤其在任務困難時，解釋反而可能讓使用者更快接受建議而不加思索——因為使用者是在權衡「自己動腦驗證的成本」與「直接採信的成本」。這代表可解釋性介面不能只回答「為什麼這樣建議」，還需要同時降低使用者「動腦驗證」的成本。

這兩個概念直接對應本階段要設計的 Agent 觸發規則：Agent 的行動階段要保留人工核可節點（對應演算法反感對策——保留最終控制權），Agent 的思考階段要能誠實表態信心程度（對應過度依賴悖論對策——讓使用者知道何時該提高警覺，而不是靠一段解釋文字讓使用者「感覺安心」但沒有真的降低驗證成本）。

操作步驟

  1. 為每個 Agent 功能畫出「感知（偵測到什麼情境）→ 思考（判斷該不該行動、信心程度多高）→ 行動（執行什麼、或呈現什麼選項）」三段式流程圖。

  2. 針對「思考」階段，明確定義信心分數的高／中／低門檻，並規定低信心時一律轉為「詢問使用者」而非「自動執行」。

  3. 針對高風險或不可逆的「行動」，加入人工核可節點（例如彈出確認視窗），不允許 Agent 全自動執行。

  4. 額外練習：針對自己設計的 Agent 功能，具體寫下「如果這個 Agent 這次判斷錯了，使用者下一步可以怎麼修正」，確認答案不是「沒辦法，只能重來」。

若信心分數為高（例如使用者在同一個測驗題目連續錯 3 次）：Agent 直接主動彈出「需要複習這個章節嗎？」的建議卡片，附上具體依據（「你在第 3 題已經錯了 3 次」），並保留「不用了，我自己再想想」的否決選項。

5-2｜Design-to-Code 自動化轉譯與程式碼優化

這一節把前面學的工具收斂成單一流程：設計讀取、程式碼生成、審查優化，全部在 Claude Code 一個介面完成，不用再開三個工具切來切去。最後我會介紹兩種互補的驗證方法——LLM 原型法跟 Wizard of Oz 原型法——為接下來的部署前驗證關卡做準備。

核心內容

過去這一步驟常需要用到 Builder.io 做語意對應、Cursor AI 做程式碼審查，等於一個流程要在三個工具間切換。這裡收斂到 Claude Code 一個介面完成，因為 Code Connect 已經確保生成的程式碼直接引用真實存在的元件庫，不需要再靠額外工具做語意對應；而程式碼審查與結構優化（讀取、重構、寫測試、迭代）本來就是 Claude Code 的原生能力。整條路徑從設計到程式碼都由同一個 AI agent 透過 MCP 貫通，讓「單一意圖驅動流程」取代「多工具拼接」。

核心內容：AI 設計原型法（銜接下一節）

LLM 原型法（用真實模型測試可行性）與 Wizard of Oz 原型法（略去真實 AI、由人操控，純測期望性）是兩種互補的驗證方法。實務上建議依專案不確定性高低決定順序：概念極不確定時，先用 Wizard of Oz 測試「就算做到完美，使用者也真的想要」；概念已初步驗證但不確定技術能否達成時，才用 LLM 原型法測試現有模型是否真的能穩定產出期待的品質。第五週已經用真實模型把 Agentic 功能做出來了（可行性已驗證），部署前驗證關卡要反過來，用 Wizard of Oz 精神再測一次「使用者是否真的想要這個自動化體驗」，作為部署前的最後把關——避免把資源投入到一個技術上做得到、但使用者根本不需要或反感的功能上。

操作步驟

  1. 確認 Code Connect 已對應好第四週建立的元件庫，讓 Claude Code 生成的程式碼直接引用真實存在的元件，不需額外做語意對應。

  2. 直接請 Claude Code 執行程式碼審查與結構優化（讀取、重構、寫測試、迭代），不需切換到其他 IDE。

  3. 全程只在 Claude Code 一個介面完成「設計讀取 → 程式碼生成 → 審查優化」，體會「單一意圖驅動流程」而非多工具拼接的差異。

  4. 額外練習：回顧第五週做出來的 Agentic 功能，先自問「這個功能的可行性，我驗證過了嗎？」（應該可以肯定回答，因為是用真實模型做出來的），再自問「這個功能的期望性，我驗證過了嗎？」（這正是部署前驗證關卡要處理的缺口）。

部署前驗證關卡（上線前的最後把關）

驗證關卡一｜Wizard of Oz 測試設計

部署之前，先停下來回答一個問題：這個功能就算做到完美，使用者真的想要嗎？注意，這題問的不是「技術做不做得到」，是「有沒有人要」——兩題常常被混在一起，混在一起的代價通常很貴。這一節你會設計並執行一場 Wizard of Oz 測試：由人在幕後扮演 AI，讓回覆品質「永遠完美」，純粹測期望性。

核心內容

Wizard of Oz 原型法讓受測者以為自己在與一個真實運作的 AI 系統互動，實際上系統的核心判斷由操控者在幕後即時手動操控完成。這個方法的特殊之處，在於它刻意「略去實際的 AI 元件」：回覆品質由操控者的判斷力決定，因此可以做到「品質永遠是完美的」，藉此純粹測試「如果這個功能完美運作，使用者會不會想要、會不會採用」，而不受目前模型技術能力高低的干擾——這跟第五週用真實模型做出來的 Agentic 原型，測的是完全不同的問題。

操控者角色設定有四個原則：人設一致性（維持單一語氣風格，避免忽冷忽熱）、回應延遲控制（加入 1–3 秒思考延遲，模擬真實感）、不主動暴露操控痕跡（避免「我幫你問一下」這類真人用語）、允許適度的不完美（偶爾給出不夠精確的答案，更貼近真實 AI 行為，才能觀察受測者的追問與修正行為）。

操作步驟

  1. 找 2–3 位同事或朋友扮演使用者，針對剛完成的 Agentic 功能設計 2–3 個任務腳本（例如：「請這個自動化助教幫你安排下一步學習內容」）。

  2. 自己或同組另一人扮演操控者，在受測者看不到的畫面幕後即時回應，模擬 Agent 已經完美運作的假象。可套用以下 Prompt 範本，讓 Claude 生成回應草稿，操控者審核後再手動轉發：

    xxxxxxxxxx
    角色設定：你是 [產品名稱] 的 AI [角色名稱]，服務對象是 [目標使用者]。
    語氣與風格：親切但專業，每次回答控制在 3-5 句話內；遇到不確定的問題，
    誠實表示「這部分我需要再確認」，不要編造內容。
    目前情境：使用者正在 [情境描述]，任務是 [任務描述]。
    使用者提問：[貼上受測者的即時提問]
    請生成一則以此角色身分回覆的草稿，供操控者審核後手動發送給使用者。

  3. 記錄行為觀察指標：任務完成度、對話輪數、誤觸或困惑次數、是否要求「真人協助」，判斷「就算做到完美，使用者是否真的想要」。

  4. 額外練習：測試結束後追加一題「你覺得剛剛是不是真人在跟你對話？」，觀察操控是否有明顯破綻，作為下一輪測試的改進依據。

驗證關卡二｜介面設計與模型需求轉譯

測出來的需求，要落回介面跟規格才算數。這一節你會把上一節的觀察轉成需求對應矩陣與結構化線框圖，再盤點 AI 模型能力需求規格，跟你手上的 Agent 逐項比對——缺什麼，就是部署前的待辦清單。

核心內容

驗證關卡一觀察到的使用者需求，有一個共通點：使用者不一定擅長「下指令」。若介面只提供一個空白對話框（純提示詞介面），會讓求助意圖轉化為表達能力的門檻，這是「Spork 問題」的典型徵狀。因此這一步採混合式介面：以自由文字輸入為基礎，同時搭配結構化的快速選項、卡片與操作按鈕。

要讓介面能呈現步驟卡、推薦卡等結構化元件，模型必須具備「結構化輸出能力」——能依問題類型輸出對應格式（純文字／步驟清單／推薦卡片），供前端渲染成對應元件。這是避免 Spork 問題的技術前提：若選用的模型不支援可靠的結構化輸出，畫面就無法呈現卡片式互動，屆時建議退而求其次，在前端以規則式樣板依關鍵字分類。

操作步驟

  1. 把驗證關卡一觀察到的使用者需求，套進「需求對應矩陣」：每個痛點對應一個互動設計原則與介面元件（例如：「問題不清楚」→「情境感知快速提問卡」；「卡在步驟」→「分步驟卡片＋下一步按鈕」）。

  2. 畫出（或用 Figma／Stitch 快速產出）對應的結構化線框圖：快速提問卡、分步驟卡片、推薦卡片、不確定／升級出口，避免整個功能只剩一個空白對話框。

  3. 條列 AI 模型能力需求規格（上下文長度、多輪對話狀態、RAG、結構化輸出、不確定性表態、回應延遲），對照 Claude Code 目前產出的 Agent 是否都已滿足，缺口列為部署前待辦事項。

  4. 額外練習：檢查自己設計的「不確定／升級」畫面，是否明確表態「這部分我不確定」並提供至少兩個結構化出口（例如換個問法、轉真人協助），而不是讓使用者卡住只能默默放棄。

第六週：網站部署實務與域名營運

6-1｜基礎網站部署知識與環境選型

從這一節開始，把成果變成一個真正上線的網站。你會先弄懂靜態託管跟伺服器渲染（SSR）差在哪——這決定你該選哪個平台；然後從 Vercel、Netlify、GitHub Pages 挑一個，完成本課程（也可能是你人生）的第一次部署。

核心內容

買過便當嗎？便當店有兩種：一種是便當裝好排在檯面上，你來了直接拿，快又便宜；另一種是現點現炒，你點了老闆才下鍋，等久一點，但每個人拿到的都不一樣。靜態託管就是第一種——網站內容在建置時就做好固定的 HTML 檔案，伺服器只負責原封不動地端出來；伺服器渲染（SSR）是第二種——每次有人來，伺服器現場運算出屬於他的頁面。作品集、行銷頁這種人人看到都一樣的內容，拿現成便當就好（速度快、成本低）；要登入、要顯示個人化內容的產品，就得現點現炒（SSR 或混合渲染）。

工具介紹｜Vercel／Netlify／GitHub Pages 三平台比較

三者共通點是都走「連結 Git 儲存庫、push 即自動部署」的模式，差異主要在動態渲染能力與商用條款。選型時先回答「內容會不會依使用者變化」，再看免費額度的商用限制——例如把商業產品放在 Vercel Hobby 方案上是違反服務條款的，正式營運需升級付費方案。

註：以上功能與定價為撰寫當下資訊，工具介面、方案與額度可能隨版本更新而變動，請以官方最新版為準。

操作步驟

  1. 釐清產品屬性：內容是否會頻繁動態變化（傾向 SSR）還是大致固定（傾向靜態託管）。

  2. 依此比較 Vercel、Netlify、GitHub Pages 三者在部署流程、免費額度、與 CI/CD 整合度上的差異。

  3. 選定一個平台，完成帳號註冊與第一次手動部署，確認網站可透過平台提供的預設網址開啟。

6-2｜域名申請與 DNS 註冊流程實作

網站上線了，但網址還是平台送的那串 xxx.vercel.app，跟開店用別人的招牌差不多意思。這一節你會買下自己的網域，弄懂 A Record 跟 CNAME 差在哪，然後把自訂網域跟部署平台綁起來。

核心內容

網域註冊商跟部署平台是兩家不同的店，中間靠 DNS（網域名稱系統）牽線。DNS 記錄有兩種常用寫法：A Record 像是把你的店名登記到一個固定的門牌號碼（IP 位址）；CNAME 像是登記成「跟著某某店走」（指向另一個網域）。多數現代平台建議用 CNAME，理由很實際：平台底層的 IP 會搬家，你登記死門牌，哪天它搬走你的網站就失聯了；登記「跟著它走」，它搬到哪你跟到哪。

工具介紹｜Namecheap／GoDaddy（網域註冊商）

註：以上功能與定價為撰寫當下資訊，工具介面、方案與額度可能隨版本更新而變動，請以官方最新版為準。

操作步驟

  1. 在 Godaddy 或 Namecheap 搜尋並購買一個符合產品品牌的網域名稱。

  2. 進入網域註冊商的 DNS 管理頁面，新增 A Record（指向平台 IP）或 CNAME（指向平台網域），依所選部署平台官方文件填入對應數值。

  3. 等待 DNS 傳播（通常數分鐘至數小時），用自訂網域開啟網站確認綁定成功。

  4. 額外練習：用線上 DNS 查詢工具（例如 whatsmydns.net）查詢自己設定的記錄是否已在多個地區的 DNS 伺服器上生效，理解「DNS 傳播」不是瞬間發生的。

6-3｜安全性與自動化運維

正式上線前的最後兩塊拼圖：HTTPS 加密跟 CI/CD 自動部署。做完這一節，你會親眼看到「設計一動、網站同步」——push 一次程式碼，網站自己建置、自己更新，全程不用手動上傳任何檔案；那個瞬間的爽感，很難用文字形容，你自己體會。

核心內容

HTTPS 加密不只是提升安全感，也直接影響 SEO 權重與瀏覽器是否會顯示「不安全」警示，是正式上線產品的基本門檻。CI/CD（持續整合／持續部署）則是本課程「設計一動，網站同步」理念的最後一塊拼圖：當 GitHub Repository 與部署平台連動後，每次程式碼推送到主分支，平台會自動重新建置並發布，不需要每次手動上傳。

工具介紹｜GitHub

註：以上功能與定價為撰寫當下資訊，工具介面、方案與額度可能隨版本更新而變動，請以官方最新版為準。

操作步驟

  1. 在部署平台後台一鍵開啟 HTTPS／SSL（多數平台會自動簽發並更新憑證）。

  2. 將專案的 GitHub Repository 與部署平台連動，設定「push 到主分支即自動部署」。

  3. 實測一次：在 Figma 或程式碼中做一個小改動並 push，確認網站內容確實同步更新，驗證「設計一動、網站同步」的自動化發布是否成立。

  4. 額外練習：刻意 push 一個會導致建置失敗的錯誤程式碼，觀察平台的建置失敗通知與錯誤訊息，熟悉排查建置錯誤的基本流程。

老實說，這門課教的工具，過兩年大概會換掉一半（覺得自己講得還太保守）。所以重點從來不是記住每個工具的操作步驟，而是那幾條不會過時的判斷原則：
